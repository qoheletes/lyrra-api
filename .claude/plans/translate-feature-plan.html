01 Current State of Translation

Before choosing a direction, it helps to map what already exists and where the seams are.

What exists today

Endpoint

POST /youtube/translate

Downloads YouTube audio, sends it to OpenAI Whisper with task="translate", caches the JSON result, returns English text + segments.

Storage

File cache + DB

Full transcription JSON cached in data/transcriptions/. DB stores translated_text, translated_segments (JSONB), and translated_language on the videos table.

Subtitles

SubtitleTrack model

Already supports multi-language VTT upload/serve/delete with a unique constraint on (video_id, language_code). Has is_machine_translated flag.

Hard limitations today

Gap 1

English-only output

Whisper's task="translate" always targets English. There is no path to generate subtitles in Spanish, French, Japanese, etc.

Gap 2

No source-language subtitles

Using task="translate" discards the original language. A Korean video gets English only — no Korean subtitles are ever stored.

Gap 3

No auto-VTT generation

The pipeline returns segments but never automatically creates a SubtitleTrack + VTT file from them. Those are uploaded manually.

Gap 4

No async / background jobs

Everything runs synchronously in the request. Long videos block the HTTP response with no progress updates.

02 Four Viable Directions

Each option is additive — later options build on earlier ones — so the real question is how far to go in one sprint.

Recommended

A — Multi-Language Subtitle Generation

After the existing English transcription is stored, call a text translation API (OpenAI GPT or DeepL) on the English segments to produce translated segments, then render them into VTT files and save SubtitleTrack rows automatically.

Core idea: Whisper → English segments → GPT/DeepL translate → VTT files per target language → SubtitleTrack records.

Pros

  • Directly fills Gap 1 and Gap 3
  • Reuses the existing SubtitleTrack model as-is
  • Minimal new schema (one new table)
  • GPT translation handles context better than word-by-word Whisper
  • Segments already have timestamps — VTT is trivial to render
  • Separates concerns: transcription vs. translation

Cons

  • Second API call cost per target language
  • Translation quality depends on English Whisper quality as input
  • Still no native-language subtitles (source language discarded)
Alternative

B — Dual-Pass: Transcribe + Translate

Run Whisper twice per video: once with task="transcribe" (preserves source language) and once with task="translate" (English). Store both. This gives native-language subtitles for free.

Pros

  • Closes Gap 2 — native language subtitles available
  • Source text can seed higher-quality multi-language translation
  • No new translation API needed for source language

Cons

  • 2× Whisper API cost on every new video
  • Doubles processing time per request
  • Source language may be unknown until first pass completes
  • More DB schema changes needed to store source segments
Alternative

C — Background Job Queue

Decouple the HTTP response from processing. Introduce a job model — the endpoint returns a job_id immediately, and processing happens async. Clients poll or receive webhooks when subtitles are ready.

Pros

  • Solves Gap 4 — no more blocking long videos
  • Enables parallel multi-language jobs
  • Better UX for a frontend

Cons

  • Significant infra change (Celery/ARQ/asyncio workers)
  • Breaks existing synchronous client contracts
  • New table, new polling endpoint needed
  • Overkill if videos are typically under ~10 min
Complex

D — Streaming / Real-Time Translation

Stream translated subtitle cues to the client as they are generated (SSE or WebSocket), allowing a player to show subtitles in near-real-time as the video is first processed.

Pros

  • Best perceived performance
  • No polling needed

Cons

  • Much higher implementation complexity
  • Whisper API is batch — no streaming audio-to-text today
  • Cannot do sentence-level context translation incrementally
  • Almost no benefit if results are cached after first run

04 Recommended Path: Option A

💡

The plan: After the Whisper translation completes and English segments are cached, offer a new endpoint to request translation into a target language. Use OpenAI GPT (already a dependency) to translate segments, render a VTT file, store it via the existing storage client, and create a SubtitleTrack row. Option B (dual-pass) can be added later as an incremental improvement without changing the core architecture.

Translation Provider Choice

Recommended

OpenAI GPT (gpt-4o-mini)

Already a dependency. Handles context across a full subtitle block much better than per-sentence APIs. Supports system prompts for style constraints (e.g. "keep it terse for subtitles"). Token cost for typical video: ~$0.01–0.05.

Alternative

DeepL API

Better quality for European languages, lower cost per character, but requires a new API key + dependency. Does not support system-prompt-level context control. Good swap-in if GPT quality proves insufficient for specific language pairs.

Translation Strategy: Batched Segments

Rather than translating one sentence at a time (expensive and loses cross-sentence context), we send all sentences for a video in a single structured prompt. The model returns a JSON array of translations preserving the original index order, so timestamps stay aligned. 

# Prompt structure (system message)
"""
You are a subtitle translator. Translate each item in the JSON array to {target_language}.
- Preserve meaning; favor clarity over literal translation.
- Keep translations concise — subtitles must be readable on screen.
- Return a JSON array of strings in the same order as the input.
- Do not add explanations or extra text.
"""

# User message
"""Translate these subtitles to {target_language}:
{json.dumps([seg["text"] for seg in segments])}
"""

VTT Rendering

Segment timestamps are already stored in milliseconds. The renderer is straightforward:

def render_vtt(segments: list[dict], translated_texts: list[str]) -> str:
    lines = ["WEBVTT\n"]
    for i, (seg, text) in enumerate(zip(segments, translated_texts)):
        start = ms_to_vtt_time(seg["start"])
        end   = ms_to_vtt_time(seg["end"])
        lines.append(f"{i+1}\n{start} --> {end}\n{text}\n")
    return "\n".join(lines)

def ms_to_vtt_time(ms: int) -> str:
    h, remainder = divmod(ms, 3_600_000)
    m, remainder = divmod(remainder, 60_000)
    s, ms        = divmod(remainder, 1_000)
    return f"{h:02d}:{m:02d}:{s:02d}.{ms:03d}"

05 Schema Changes

The videos and subtitle_tracks tables require no changes. One new table is needed to store per-language translated segments.

New table: subtitle_translations

Caches translated segment text per video/language so we don't re-call the translation API on repeat requests for the same language.

id BigInteger PK auto-increment
video_id String FK → videos.id, ON DELETE CASCADE
language_code String(10) BCP-47 (e.g. "es", "ja", "fr-CA")
translated_segments JSONB Array of {start_ms, end_ms, text} with translated text NEW
model_used String e.g. "gpt-4o-mini" — for audit and re-generation NEW
created_at DateTime server_default=now()

Unique constraint on (video_id, language_code) — same pattern as subtitle_tracks, one cached translation per language per video.

06 API Design

New endpoints

Method Path Description Module
POST /videos/{video_id}/translate Translate cached English segments to a target language, generate VTT, upsert SubtitleTrack app/translations/

POST /videos/{video_id}/translate — Request

{
  "target_language": "es",             // BCP-47 language code (required)
  "model": "gpt-4o-mini",           // optional, defaults to gpt-4o-mini
  "force_retranslate": false,       // optional, re-call API even if cached
  "generate_subtitle_track": true   // optional, auto-create SubtitleTrack + VTT
}

  "video_id": "yt_dQw4w9WgXcQ",
  "target_language": "es",

  "model_used": "gpt-4o-mini",

08 Implementation Phases

Organized as four sequential phases, each independently shippable.

1
Foundation — domain + DB layer ~2–3 hours
  • Write Alembic migration for subtitle_translations table
  • Create app/translations/domain/models.py: @dataclass SubtitleTranslation
  • Create app/translations/domain/exceptions.py: NoSourceSegments, UnsupportedLanguage, TranslationFailedError
  • Create app/translations/infrastructure/orm_models.py: SubtitleTranslationORM
  • Create app/translations/infrastructure/repository.py: TranslationRepository with find, upsert, delete, _to_domain()
  • Create app/translations/infrastructure/vtt.py: render_vtt() and ms_to_vtt_time() helpers
  • Unit tests for render_vtt() with edge cases (empty text, long cues)
2
Application layer + API endpoints ~2–3 hours
  • Create app/translations/application/ports.py: Translator(Protocol) — structural typing, no inheritance required
  • Create app/translations/application/use_cases.py: TranslateVideo, GetCachedTranslation with execute()
  • Create app/translations/infrastructure/translator.py: GptTranslator implements Translator port
  • Create app/translations/infrastructure/schemas.py: TranslateRequest, TranslateResponse, TranslatedSegment (Pydantic)
  • Create app/translations/infrastructure/router.py: POST /videos/{video_id}/translate and GET /videos/{video_id}/translate/{lang_code}; domain exceptions → HTTP via except blocks
  • Wire router + ORM import into app/main.py
  • Add language validation: define supported language list in app/core/config.py
  • Integration test: translate a cached video, assert VTT + SubtitleTrack created
3
Caching + idempotency ~1–2 hours
  • Implement cache-hit path: if subtitle_translations row exists and force_retranslate=false, return immediately with "from_cache": true
  • Implement force_retranslate path: delete old row + VTT, re-translate, upsert SubtitleTrack
  • Handle generate_subtitle_track=false case (return segments only, skip VTT/SubtitleTrack creation)
  • Test: POST same video/language twice — second call must not re-invoke GPT
4
Hardening + README update ~1 hour
  • Validate GPT response: assert returned array length matches input length before saving
  • Retry logic: 1 retry with same prompt if GPT returns malformed JSON
  • Add SUPPORTED_TRANSLATE_LANGUAGES config list (start with 10–15 common languages)
  • Update README with new endpoints + example request/response
  • Add TRANSLATION_MODEL env var defaulting to gpt-4o-mini
ℹ️

Future Phase 5 (Option B): If source-language subtitles are needed, add a task="transcribe" pass to the existing YouTube service, store native segments in a new column on videos, and populate a SubtitleTrack for the detected source language. This does not change the translation module at all.

09 Risks & Mitigations

RiskLikelihoodImpactMitigation
GPT returns wrong array length Medium High Validate len(response) == len(segments) before saving. On mismatch, retry once, then raise TRANSLATION_API_ERROR.
Token limit exceeded for long videos Medium Medium Chunk segments into ~100-segment batches. Translate in chunks, merge results before rendering VTT.
VTT file stored but SubtitleTrack insert fails Low Medium Use a DB transaction: insert SubtitleTrack first, then upload file. On DB failure, do not store the file. On upload failure, roll back the DB row.
Unsupported language codes from clients High Low Validate against an allowlist of supported codes in the request schema. Return 422 with the supported list in the error body.
Translation quality degrades for non-Latin scripts Medium Medium GPT-4o-mini handles most scripts well. Add quality note in docs. Consider DeepL swap-in for CJK languages if feedback surfaces issues.
Concurrent duplicate requests for same video/language Low Medium The unique constraint on (video_id, language_code) causes the second upsert to fail gracefully. Handle the IntegrityError and return cached result.

Open questions to resolve before starting

Decision Needed

Which languages to support at launch?

A curated list (e.g. es, fr, de, ja, ko, pt, zh, ar, hi, it) is better than "all of BCP-47." Determines the validation allowlist and testing surface.

Decision Needed

Should translation trigger automatically on /youtube/translate?

Option: auto-translate to a default set of languages on every new video. Or keep it on-demand via the new endpoint. On-demand is safer for cost control.

Decision Needed

Is the source of truth segments in DB or file cache?

Currently both exist (translated_segments JSONB in DB and JSON files in data/transcriptions/). For the translation service, reading from DB JSONB is cleaner and avoids filesystem reads.

Decision Needed

Expose re-translation to the API caller?

The force_retranslate flag is useful for dev/admin but may be confusing or expensive if exposed publicly. Consider making it an admin-only or internal flag.
